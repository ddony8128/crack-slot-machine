# RULE SLOT — 기획 (Single Source of Truth)

This is the **canonical spec** for game content and scoring. When code and this doc
disagree, this doc wins — fix the code (or, if the spec genuinely changed, update this
doc in the same change). Cross-check any content/scoring/rule edit against this file
before shipping.

> **Why this exists.** The season set-rules (cat / vehicle / monster / combos) were
> originally built from interpretation-level plan docs, not a user-authored spec, so
> they drifted from intent (random 물류 사업, non-spec pair bonuses, missing cat-rule
> scores, ×3 vs ×4 surges). §2–§15 below are the authoritative checklist; treat every
> row as a conformance test. See [Known deviations](#known-deviations) and
> [Code↔spec mismatches](#codespec-mismatches-tracking) for current status.

## Known deviations

These intentionally differ from the checklist below (explicit user decisions):

1. **유료 주차 = 원하는 2칸 (NOT 원하는 만큼).** §8 lists "원하는 만큼"; the user overrode this
   ("너무 사기다") to **keep up to 2 vehicle cells of choice** — `min(2, #vehicles)`,
   `−30` per held cell. The checklist text is left as-is for the record; the **2칸 cap is
   the shipping rule**.
2. **첨탑 족보 강화 = 각 한 번씩 (포커 족보 + 보유 세트 보너스).** The shop's "족보 강화"
   section covers BOTH the poker hands AND every owned (current-pool) set's bonuses.
   Each upgrade is buyable **once**: positive bonuses get +50점 / ×2 (`buyHandFlat`/
   `buyHandDouble`, `buySetBonus` kind flat/double); penalty bonuses (이웃 고양이) get
   **완화만** (+50 toward 0, `buySetBonus` kind mitigate). Applied in `setBonuses` via
   `setBonusUpgrades` (threaded through RunConfig like handUpgrades); shop shows
   "…완료" once bought.

## Code↔spec mismatches (tracking)

Discovered by auditing the code against the checklist (2026-06-17). ✅=fixed, ⬜=open.

- ✅ **물류 사업**: SELECT repeated per plane (직접 고른 두 칸 교체 × 비행기 수) — kind `logiswap`, no longer random.
- ✅ **우다다다 / 점프의 달인**: each awards **+40 per cat** on top of the move (the "사라진 고양이 점수").
- ✅ **0 상승 (zero-to-seven)**: converts the **가장 왼쪽 2개의 0** (was 모든 0). Patched in 빠른 게임 too.
- ✅ **보석 셔플 (gem-shuffle)**: rerolls the **가장 왼쪽 보석 2개** (was 1개). Patched in 빠른 게임 too.
- ✅ **과일/보석 확률 증가**: ×4 (was ×3). Patched in 빠른 게임 too (cat-odds already ×4).
- ✅ **7 scores / 4 penalty**: verified already correct — 7=10/77/150/500/777, 4 penalty=−30.
- ✅ **특수 족보 multipliers**: verified already correct — 4×4 → ×3, 4×5 → ×4, 0≥3 → 규칙 1장 추가.
- ✅ Non-spec 페어 보너스 (과수원 보석상 / 고양이 택시) removed; `PAIR_RULES = []`.
- ✅ 점수표 hides 특수 족보 outside 빠른 게임; uniform 한다체; calico cat emoji distinct.
- ✅ **레거시 규칙 드리프트**: 빠른 게임/이벤트가 공용 `RULES`를 "심볼 굴러가면 노출"로 걸러 시즌 추가
  과일/보석 규칙(비타민 보충·미의 추구)이 새어들고 원조의 붉은/푸른 물들이기(combo 재분류)가 빠져 있었다 →
  id 화이트리스트 `LEGACY_RULE_IDS`(원조 30)로 동결. 오퍼와 규칙 보기가 동일 기준.
- ✅ **레거시 족보 드리프트**: 원조 색 보너스 올 레드(+250)/올 블루(+200)가 시즌 config-driven scoring
  리팩터에서 누락(상수·RED_SET/BLUE_SET만 고아로 잔존) → 레거시 전용 `legacyColorBonus`로 복원.
  과일/보석 3종·올 과일·올 보석은 setBonuses가 처리하므로 중복 없음.

# 1. 모드별 규칙 구성 방식

## 1.1 빠른 게임

빠른 게임은 기존 게임 규칙과 연출을 최대한 유지한다. (`빠른 게임 = legacy quick ruleset`)

- 기존 숫자 특수 족보 유지
- 기존 빠른 게임 연출 유지
- 빠른 게임 랭킹은 시즌마다 초기화
- 빠른 게임 랭킹은 게스트와 회원 모두 표시
- 시즌 랭킹에는 반영되지 않음

빠른 게임은 시즌 모드와 규칙이 달라도 된다. 필요하면 엔진 config를 별도로 분리한다.

**레거시 동결 (빠른 게임 = 이벤트 `/e/[slug]`).** 빠른 게임/이벤트는 **"원조 룰셋"으로 동결**한다.
공용 `RULES` 테이블에 시즌 세트 규칙이 함께 들어 있으므로, 레거시 오퍼·점수표·점수는 모두 다음 기준을
따른다(코드 SoT: `LEGACY_RULE_IDS` in `data/rules.ts`, `legacyColorBonus` in `lib/score.ts`).

- **규칙**: id 화이트리스트 `LEGACY_RULE_IDS`(30개). 시즌-1 직전 커밋(`d6842dd`)에서 레거시 백으로
  노출되던 규칙과 정확히 일치. 시즌 추가분(`비타민 보충`·`미의 추구`)과 시즌 세트 규칙(고양이/교통/
  괴물·그 외 combo)은 제외, `붉은/푸른 물들이기`는 원조에 있었으므로 포함. 오퍼(`offerRules`)와
  `규칙 보기`(ReferenceModal)가 같은 화이트리스트를 써서 항상 일치한다.
- **족보**: 원조 색 보너스 **올 레드(+250)·올 블루(+200)**를 레거시 전용으로 적용(`legacyColorBonus`).
  시즌 모드는 세트 기반 보너스만 쓰므로 색 묶음(RED/BLUE)을 적용하지 않는다.
- **동작만 조정**: 과일/보석 확률 ×4 · 0 상승(왼쪽 2개) · 보석 셔플(2개) — 멤버십은 유지(화이트리스트
  포함), 동작만 spec에 맞춤. 이 때문에 빠른 게임은 pre-season-1과 바이트 동일하지 않다.
- 위 동결로 빠른 게임 점수 산식이 원조로 돌아가 `RULESET_VERSION`을 **3**으로 올렸다(랭킹 분리).

## 1.2 일일 도전

```txt
일일 도전 규칙 풀 =
오늘 선택된 기본 규칙 세트 1개 + 숫자 세트 규칙 + A세트 규칙 + B세트 규칙
(+ A-B 조합 규칙이 존재하면 포함)
```

매일 미리 DB에 저장된 config 사용: `date_key / seed / symbol_set_ids / basic_rule_set_id / pair_rule_ids`.
A-B 조합 규칙이 없으면 아무것도 추가하지 않는다.

## 1.3 첨탑 오르기

숫자 세트를 기본 포함. 초기 규칙 풀 = 첨탑 기본 규칙 8개 + 처음 선택한 심볼 세트 규칙 중 랜덤 2개 = 총 10개.
규칙 풀은 최대 10개. 새 규칙으로 10개를 초과하면 기존 규칙 중 하나를 제거해야 한다.

## 1.4 퍼즐 모드

랜덤 규칙 선택이 없다. 퍼즐별로 정해진 규칙들이 처음부터 가방에 들어 있다
(`퍼즐 규칙 = puzzle config의 고정 availableRuleIds`). 매 스핀 랜덤 3개 중 택1 방식이 아니다.

| 물류 사업 | 비행기의 수만큼 반복하여 **직접 고른** 두 칸을 교체한다 |
| 배 크다 | 가장 왼쪽에 있는 배의 양쪽 칸은 그 배 심볼을 복사한다 |

- 이동 점수 = `symbol_moved` 이벤트 기준. 재굴림 점수 = `symbol_rerolled` 이벤트 기준.
- `물류 사업`은 비행기 수만큼 선택형 교환을 반복한다.

# Season scoring & mode economy

게임 콘텐츠(§2–§15)와 별개로, **시즌 점수 환산**과 **첨탑 경제**는 아래가 SoT다. 구현은
`lib/season/scoring.ts`(점수 공식·랭킹 합산)와 `lib/spire/config.ts`(경제 상수)에 있으며, 코드와
이 절이 어긋나면 이 절이 우선이다.

## S1. 시즌 구성

- **Season 1**: slug `2026-06-season-1`, 기간 6/15 12:00 ~ 6/28 12:00 KST.
  `CLIENT_VERSION=2.0.0`, `RULESET_VERSION=2` (`lib/version.ts`).
- **시즌 랭킹**(`/season/leaderboard`)은 한 플레이어의 `첨탑 + 퍼즐 + 일일` 시즌 점수를 합산해 정렬한다.
- 빠른 게임은 시즌과 무관(별도 랭킹, 시즌마다 초기화, 게스트·회원 모두 노출).
- 점수를 주는 제출(첨탑/퍼즐/일일)은 `SeasonScoreChange`(이전/이후 시즌 총점 + 순위 델타)를 반환해
  결과 화면이 카운트업 연출을 한다.

## S2. 퍼즐 점수

```
퍼즐 1개 클리어 점수 = 100 + (스핀 한도 − 사용 스핀) × 10        // puzzleScore()
시즌 퍼즐 점수 = 각 퍼즐의 베스트 점수 합                        // 퍼즐별 best_scores 행을 합산
```

- 클리어 여부만 판정한다(스핀별 점수 정산 없음). 미클리어는 0점.
- `best_scores`에 퍼즐 키별로 1행씩, 그 퍼즐의 최고 점수를 보관한다.

## S3. 일일 점수

```
첫 플레이 보너스 = +20 (그 날 처음 플레이하면; DAILY_FIRST_PLAY)
순위 보상(정산된 날만) = 상위 10% +50, 상위 50% +30, 그 외 0     // dailyRankReward()
시즌 일일 점수 = Σ(첫 플레이 +20 + 그 날 정산 순위 보상)
```

- 하루 도전 횟수: 기본 3회 + 광고 충전. 순위 보상은 **정산된(settled) 날**만 부여되며, 정산 시점에
  각 일일 행의 `seasonPoints`로 lazy-persist 된다(`settleDueDailyChallenges`).
- KST 12:00 경계로 날짜(date key)가 바뀐다. 모두가 같은 시드·규칙 풀로 플레이한다.

## S4. 첨탑 점수 (베스트 런 1개만)

```
시즌 첨탑 점수 = 최고 도달(클리어) 스테이지 × 100               // SPIRE_STAGE_POINTS
              + 남은 돈 × 10                                  // SPIRE_MONEY_PER
              + 남긴 스핀 × 10                                // SPIRE_SPIN_PER (클리어한 스테이지들의 잔여 스핀 합)
```

- 한 플레이어당 베스트 런 1개만 시즌 점수로 반영한다.
- 첨탑 자체 랭킹(`/api/spire/leaderboard`)은 `도달 스테이지 → 총 점수` 순으로 정렬, 결과 화면과
  시즌 허브에서 노출한다.

> **과거 캡 메모**: `SEASON_MODE_CAP=1000`(모드당 상한) / `DAILY_COUNT_CAP=10`(최고 10일) 상수와
> `puzzleSeasonPoints`/`seasonDailyTotal`는 v1 합산용으로 남아 있으나, 현재 `buildSeasonRanking`은
> **모드별 캡 없이** per-row `seasonPoints`를 합산한다(주석 참조).

## S5. 첨탑 경제 (v0)

스테이지 10개. 스테이지 목표 점수: 500 / 1000 / 2000 / 4000 / 6000 / 8000 / 10000 / 12000 / 15000 / 20000.
각 스테이지 최대 7스핀, 목표 도달 시 조기 클리어.

| 항목 | 값 |
|---|---|
| 시작 돈 | 0 |
| 최종 실패 | 누적 3회 실패 시 런 종료 (`SPIRE_MAX_FAILURES`) |
| 심볼 주머니 총량 | 항상 20 (`SPIRE_BAG_TOTAL`); 시작 0×12·4×5·7×3 → 세트 선택 시 0 −3, 고른 세트 3심볼 +1씩 |
| 규칙 풀 상한 | 10 (초과 시 기존 규칙 1개 제거; 기본 8 + 첫 세트 규칙 랜덤 2) |
| 클리어 정산 | 이자 = floor(정산 전 잔액 / 5) · 남은 스핀당 +2 · 클리어 보수(st1–3=4 / st4–6=6 / st7–9=8 / st10=0) |
| 비최종 실패 지원금 | +5 (`SPIRE_FAIL_SUPPORT`) |
| 아티팩트 선택 | 스테이지 3·6·9 클리어 시 |

**상점 가격**: 아티팩트 슬롯 1~3 = 6/5/4 · 세트 추가 = 3 · 규칙 추가 = 1 · 족보 강화(+50) = 1 ·
족보 강화(×2) = 3 · 리롤 = 1(차임벨 보유 시 상점마다 2회 무료) · 심볼 +1 = 그 심볼의 현재 개수(체증).

**족보 강화(once each)**: 포커식 족보뿐 아니라 **현재 보유한 모든 심볼 세트의 보너스**도 강화 대상이다
([Known deviations](#known-deviations) §2). 양수 보너스는 +50점/×2를 각각 한 번씩, 페널티 보너스
(이웃 고양이)는 **완화만**(0에 가까워지는 +50) 한 번. `setBonusUpgrades`로 RunConfig에 실려 `setBonuses`에
적용되고, 구매 후 상점은 "…완료"로 표시한다.

## S6. 영속화 · 이어하기

- 첨탑 런은 **스테이지 시작·종료 시 행동 로그를 서버에 자동 저장**한다(`POST /api/spire/progress` →
  `game_runs.actions`, status `pending`). 새 마이그레이션 없이 기존 테이블을 재사용한다.
- 시즌 허브 카드와 첨탑 진입 시 `GET /api/spire/current`로 진행 중 런을 찾아 **이어하기**를 제공한다
  (서버가 `replaySpireRun`으로 도달 스테이지를 재계산).
